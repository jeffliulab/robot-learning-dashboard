# 机器人关节图制作指南 · Robot Learning Dashboard

> 2026-07-10 定稿。本指南专管一类图：**画着机器人骨架、标注关节/控制量的手绘 SVG**（任务示意图、控制量示意图）。
> 它是《实验报告模板.md》的配套细则——模板管"报告里必须有什么图、图注怎么写"，本指南管"这类图具体怎么画才不翻车"。两者冲突时以模板为准。
> 定稿范例：G1 平地行走报告的图 3（`content/views.js` 的 `g1ControlSVG()`）；反面教材见文末（同一张图的两版废稿，真踩过的坑）。

---

## 0. 这类图存在的目的（先想清楚再动笔）

一张关节图只回答三个问题，按顺序：

1. **系统长什么样**——机器人的部位构成（头/躯干/臂/腿），读者一眼认出"这是个人形/四足"。
2. **能操纵什么**——受控关节在哪、一共多少个、控制量是什么形式（位置目标还是力矩）。
3. **控制量怎么变成运动**——因果链的最后一环（如"位置目标 → 隐式 PD → 力矩"）。

动笔前先列一张**信息点清单**：这张图必须传达哪几条事实（每条都要能在真实 cfg 里指出来源）。清单之外的东西一律不画——上一版翻车的直接原因就是想"顺便"表达太多，四条分发箭头把图搅成了乱麻。

---

## 1. 数字红线：关节数必须可自证

- **关节数量来自真实配置**，不许凭印象写。G1 minimal 的 37 = 腿 6×2 + 躯干 1 + 臂 5×2 + 手指 7×2，逐项数的是 `G1_MINIMAL_CFG` 实际驱动的关节。画图前先把这笔账算平。
- **图内标注要让读者自己能加出总数**。定稿的做法：分组引线标「腿 ×6/侧 · 臂 ×5/侧 · 手指 ×7/侧 · 躯干 ×1」——6×2+5×2+7×2+1=37，图形自证，和正文"37 维动作"闭环。
- **圆点 = 关节组，不是关节本身**。骨架上画不下 37 个点，就把同部位关节并作一点（髋部 3 个自由度并作一点），但必须在图顶注记里说明这一约定，并给每个点加 `<title>`（鼠标悬停显示关节名和自由度数，如 `left hip ×3`）。**不许画一堆点又不说每个点代表什么。**
- **反直觉的数字必须当场给分解**（2026-07-10 实战教训）：「手指 ×7/侧」让 Jeff 当场愣住——人手五指，7 从哪来？答案是 G1 装的是**三指**灵巧手：拇指 3 + 食指 2 + 中指 2 = 7（官方资产的关节名还是无语义的 `zero`–`six`，雪上加霜）。凡是读者第一反应会"咦？"的数字，必须在正文、图注或悬停 title 里给出分解式，不能只甩一个总数。核对这类数字的两条路：Isaac Lab 的 reward/actuator cfg 里的 `joint_names` 模式清单（如 `rough_env_cfg.py` 的 `joint_deviation_fingers` 正好 7 个模式）；menagerie 同型号 XML 的实名关节（`thumb_0/1/2`、`index_0/1`、`middle_0/1`）。

## 2. 骨架怎么画（人形火柴人配方）

**视角**：正视（frontal）最不容易歧义——双臂双腿对称展开，部位一目了然。写清"正视示意"。侧视只在必须表达前后关系时用（如 Ant 单腿放大图）。

**比例纪律**（血泪教训：手和膝挤在一起 = 读者认不出哪是手哪是腿）：

- 部位分区：**臂在上半区、腿在下半区**，手爪末端与膝盖至少水平错开 ≥ 20px。
- 手臂要"外展"：肘、腕逐级向外，别让手垂到大腿旁边。
- 定稿 G1 骨架的实测坐标配方（viewBox 980×350，可整体平移复用）：

```
头        <circle> 圆心 (620, 50) r=17
躯干      M620 67  L620 150                      粗 7
双臂      M620 92  L572 145 L548 205（左，右侧镜像 668/692）   粗 5
手指      腕点向下三根短须，如 M548 205 L536 225 …             粗 3
双腿      M620 150 L590 222 L590 300（左，右侧镜像 650）        粗 5
关节组点  <circle class="ctl" r=5>，位置取在线段端点或线段中点
```

- 关节组点位（13 个）：躯干 (620,120)、肩 ×2（臂线上 t≈0.3 处）、肘 ×2（臂折点）、腕/手 ×2（臂末端）、髋 ×2（大腿线中点）、膝 ×2（腿折点）、踝 ×2（腿末端）。

**四足**参照 Ant 的两张图（`antSVG` 俯视 + `antLegControlSVG` 侧视放大一条腿）：俯视讲整体布局，侧视放大一条腿讲控制量，其余腿"同构"一句话带过。

## 3. 控制量与力矩的记号（模板红线，此处给操作细节）

- **力 → 直箭头**，标注形式化记号（如 F）。
- **力矩 → 绕关节的弧形箭头 ⟳**：用 `A` 圆弧命令绕着关节点画约 3/4 圈 + 箭头 marker，τ 标签放弧的**外侧**（远离骨架一侧）。示例挑 1–2 个**周围空旷**的关节（定稿选双肘——手腕/膝盖那一带太挤，放弧必然纠缠）。
- **位置目标类控制量**（G1 的 qᵢ*）：不要给每个关节画箭头！用「一根主箭头指向机器人 + 分组引线标数量」表达"37 个目标分发到全身"。⛔ **禁止**从半空拉多条曲线戳向不同关节——那是上一版最大的败笔：像外力打在身上，且 4 条线对不上 37 这个数。
- **任务图与控制图分开**（模板原话）：任务图讲"要去哪"（速度命令箭头），控制图讲"能操纵什么"（关节+力矩）。两张图里最抢眼的箭头不得是同一根。

## 4. 文字排版（同《实验报告模板.md》「手绘 SVG 排版纪律」，此处只列要点）

- **先算字宽再定坐标**：1 汉字 ≈ 1 字号宽（.sb=12px、.lb=13px），拉丁/数字 ≈ 0.58 字号，上下标 ≈ 0.45，空格 ≈ 0.32；按 text-anchor 从锚点向两边展开。15 个汉字 = 180px，凭感觉放必撞。
- 文字离笔画 ≥ 20px；整行落在 viewBox 内、四边留 ≥ 30px。放不下**加宽画布**，不许硬塞。
- 注记分区：视角/约定类注记放画布顶部留白带；成列说明文字放左右专用空栏。
- 引线标注用 `class="eg"` 灰细线（机器检查对 eg 免检，因为它天然贴着文字）；标签锚点选 end（左侧标签）或 start（右侧标签），让文字**背离**骨架方向展开。
- 刻意画在形状里的字（框内标题、圆内"躯干"）：机器检查自动放行"完全在形状内"，判不了的加 `data-in`。

## 5. 站内固定件（直接用，别自造）

| 类/元素 | 用途 |
|---|---|
| `.nd` / `.nd.io` | 节点框/被强调节点（accent 底色）——Actor、关节位置目标框、头部圆 |
| `.ar` | 黑实线笔画：骨架线（加大 stroke-width）、流程箭头、力矩弧 |
| `.eg` | 灰色细引线（指向文字标注） |
| `.lb` / `.sb` | 13px 加粗标签 / 12px 次级说明 |
| `.ctl`（图内 `<style>` 定义） | 橙色关节组圆点（accent 填充 + paper 描边），配 `<title>` 悬停 |
| `<marker>` 箭头 | 每张图用独立 id（如 `mkG1Ctl`），防多图同页 defs 串号 |

配色只用 CSS 变量（`var(--ink)` / `var(--accent)` / …），不写死色值——站点亮暗主题都要能看。

## 6. 制作流程（每张图都走一遍）

1. **列信息点清单**，每条注明真实来源（cfg 字段/实测值）。
2. **画布分区草图**：骨架放哪、文字列放哪、注记带在哪，按第 4 节公式核算每行字的占位宽度。
3. 写 SVG 字符串函数（照 `g1ControlSVG()` 的组织：注记 → 流程框 → 骨架 → 关节点 → 分组标注 → 文字列 → 力矩弧）。
4. **跑机器检查**：`node scripts/check_report.js`——第 [8] 项抓文字压图形/越出画布。
5. **截图亲验**（机器全绿 ≠ 好看，中间版本就是全绿但手膝仍挤成一团）：
   用 headless Chrome 把这张图单独渲染出来看——写个临时脚本（放 scratchpad/temp，用完删）：用 node `vm` 跑 `content/views.js`（stub `window`、先加载 `net-diagram.js`），调用目标 SVG 函数拿字符串，包一层引 `assets/app.css` 的 HTML，然后
   `google-chrome --headless=new --screenshot=out.png --window-size=1200,480 file://.../fig.html`，肉眼检查：部位认得出吗？数字对得上吗？箭头语义唯一吗？
6. 不满意回到第 2 步迭代；满意后整页 check_report 全过 + dev 服务器硬刷新终验。

## 7. 反面教材（2026-07-10，G1 图 3 两版废稿）

**废稿 1 的三宗罪**（概念链全对，图形表达全错——"语义对"救不了"画得乱"）：

1. 骨架失真：手臂太低太靠内，手爪和膝盖同高相邻，整体像四条腿的虫。
2. 四条橙色曲线从半空戳向 4 个随机关节：想表达"目标分发"，读出来是"外力抽打"，还和 37 对不上。
3. 灰色虚线小弧当力矩：看不出绕哪个关节转，膝部的弧连 τ 标签都没有。

**废稿 2 的教训**：修完机器检查全绿，但截图一看手爪还是贴着大腿、力矩弧和右手纠缠、τᵢ 标签悬在腕膝之间归属不明。→ **第 [8] 项检查只保证"字不压图"，"图能不能看懂"必须截图用眼睛验**，这就是流程第 5 步存在的原因。

**定稿改法**（对照着记）：手臂外展让手膝水平分开 → 乱箭头换成"一根主箭头 + 分组数量引线（可加总自证 37）" → 力矩改绕双肘的 ⟳ 弧 + 外侧 τᵢ 标签 → 顶部注记写明"圆点 = 关节组"约定。
